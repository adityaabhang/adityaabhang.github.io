Overview

This group project integrated three physically separate robotic systems — two Techman cobot arms and a Fetch Robotics autonomous mobile robot (AMR) — into a single, seamlessly coordinated pick-and-place and material transport workflow executed without human intervention between steps.

The challenge was not just programming each robot independently, but making them communicate and hand off tasks to one another in the correct sequence. Node-RED served as the central orchestration layer: using Modbus TCP to command the cobot arms and HTTP webhooks and triggers to interface with FetchCore, the AMR's cloud fleet management platform.

6-step automated workflow: Daisey picks up a cup → Jerry transports the cart to Rosie → Rosie loads a magnet into the cup → Jerry returns the cart → Daisey unloads the cup back to origin. Zero manual steps between tasks.

The Robots

Daisey (TM-12S): A high-speed Techman collaborative robot arm with 12 kg payload and 1300 mm reach. Equipped with an AI-driven vision system. Handled both the initial cup pick-and-place from table to cart, and the final unload back to the table at the end of the cycle.

Rosie (TM-12): A Techman TM-12 cobot sharing the same payload and reach as the TM-12S but designed for flexible, adaptive tasks. Used a vacuum pump end-effector to pick up a magnet and precisely drop it into the cup on the cart while Jerry held position at the Rosie station.

Jerry (Freight100): A Fetch Robotics AMR with a 100 kg payload capacity and real-time obstacle avoidance navigation. Responsible for all material transport — docking under the cart, carrying it from Daisey to Rosie and back, and departing autonomously after each handoff.

System Workflow

Task 1 — Daisey picks up the cup: The cycle begins when an operator clicks the start button in Node-RED. This fires a Modbus request to TM-12S, triggering its pick-and-place routine to lift a cup from the table and set it into the cup holder on the cart.

Task 2 — Jerry docks under the cart: On Daisey's completion signal, Node-RED fires an HTTP trigger to FetchCore, instructing the Freight100 AMR to navigate autonomously under the cart and pick it up for transport.

Task 3 — Jerry transports to Rosie: Jerry navigates to the TM-12 workstation. FetchCore sends a webhook back to Node-RED on arrival, triggering the next step.

Task 4 — Rosie loads the magnet: Rosie's vacuum pump end-effector picks up a magnet and precisely places it inside the cup sitting on the cart.

Task 5 — Jerry returns the cart: Another Node-RED trigger dispatches Jerry back to the original Daisey station with the loaded cart.

Task 6 — Daisey unloads the cup: Daisey executes a final pick-and-place routine, removing the cup from the cart and returning it to the original table position — completing the full autonomous cycle.

Challenges

Heterogeneous protocol bridging: The three robots run on entirely different software ecosystems — TMFlow with Modbus, FetchCore with REST webhooks — and there is no native interconnect. Node-RED had to bridge them reliably, handling out-of-order events and timing mismatches without a custom server.

Reliable task sequencing: Each of the six tasks depended on the precise completion of the previous one. A delayed Modbus status flag or a late FetchCore webhook could break the chain. Debugging required stepping through each handoff carefully and adding status confirmation checks in Node-RED before triggering the next step.

Spatial coordination in the lab: Jerry, Daisey, and Rosie each occupy fixed positions in the Build@Scale lab. The AMR's waypoint annotations in FetchCore, the cobot arm waypoints in TMFlow, and the cup holder geometry on the cart all had to align precisely for every task to succeed.

Outcomes

3
Real robots integrated: TM-12S, TM-12, Freight100
6
Automated task steps with zero human intervention between them
0
Custom middleware servers — Node-RED handled all integration

The full 6-task cycle executed autonomously end-to-end. Node-RED proved that visual flow-based programming is a practical and fast-to-debug orchestration tool for heterogeneous robotic systems — avoiding the need for a traditional middleware server or custom API layer.
